## 1. Purpose and Scope

### 1.1 Purpose

This Security Monitoring Policy establishes the mandatory requirements for continuous security monitoring, threat detection, alert triage, and security incident identification across the ABC Corp Azure cloud environment. It defines what must be monitored, how alerts must be managed, what response timelines are required, and who is accountable for each component of the monitoring program.

This policy was produced in direct response to the complete absence of a functioning security monitoring program confirmed during the March 2026 gap assessment:

- **No analytics rules (real finding):** No analytics rules were active in Microsoft Sentinel. The detection layer was entirely absent. The SecurityAlert table contained High and Critical severity alerts generated by Microsoft Defender for Cloud that had never been triaged.
- **Unactioned incidents (real finding):** The SecurityIncident table contained unactioned incidents. These incidents had never been assigned, investigated, or closed. They represent real security events in a live environment that went unreviewed.
- **Endpoint telemetry gap (real finding):** The ProtectionStatus table returned zero rows for all 126 assets. Endpoint telemetry was not flowing into Sentinel. The monitoring platform had no visibility into endpoint protection state.
- **No monitoring governance (real finding):** No monitoring coverage requirements, alert response SLAs, or escalation paths existed. No on-call rotation existed for after-hours Critical alerts.
- **SFA detection gap (real finding):** 83,600 of 203,906 sign-in events completed on single-factor authentication with no detection rule to identify this condition.

### 1.2 Scope

**In Scope:**
- All security monitoring activities within or on behalf of the ABC Corp Azure environment
- Microsoft Sentinel and the Log Analytics Workspace connected to the Azure subscription
- All data sources feeding telemetry into Sentinel (Entra ID, Microsoft Defender for Cloud, Microsoft Defender for Endpoint, Azure Activity logs, NSG flow logs, and additional connectors)
- All security alerts generated by Sentinel analytics rules, Microsoft Defender for Cloud, and Microsoft Defender for Endpoint
- All security incidents created within Sentinel
- All accounts, devices, and resources within the Azure subscription

**Out of Scope:**
- Isolated lab VMs rebuilt on each use with no persistent production connectivity (unless they generate alerts reaching production monitoring systems)
- Microsoft Azure platform-layer monitoring that is Microsoft's responsibility under the shared responsibility model
- Third-party SaaS applications not connected to the Sentinel workspace

## 4. Security Monitoring Platform Requirements

### 4.1 Microsoft Sentinel as the Monitoring Platform of Record

Microsoft Sentinel is the designated SIEM platform. All security monitoring, alert management, and incident tracking must be conducted through the Sentinel workspace connected to the Azure subscription.

A single Sentinel workspace must be maintained and kept active at all times. No security alert or incident may be deleted, suppressed, or permanently dismissed without documented review and justification. Alert suppression must be limited to confirmed false positives with documented tuning rationale.

### 4.2 Log Analytics Workspace Requirements

| Daily data cap | Configured to prevent unexpected cost overruns |

| Workspace health | Monitored weekly; ingestion anomalies investigated within 24 hours |

## 6. Analytics Rule Requirements

### 6.1 Mandatory Analytics Rule Categories

**Driving Evidence (real finding):** At time of the March 2026 assessment, no analytics rules were active in the Sentinel workspace. Every category listed below represents a detection capability that did not exist and must be built.

| Rule Category | Minimum Active Rules | Coverage Area | Priority |
|---|---|---|---|
| Break-glass account usage | 1 | Section 5.1 | Critical |
| Privileged role assignment outside PIM | 1 | Section 5.1 | Critical |
| Single-factor authentication success | 1 | Section 5.1 | Critical |
| MFA bypass or downgrade | 1 | Section 5.1 | Critical |
| Impossible travel detection | 1 | Section 5.1 | High |
| High-risk user detection | 1 | Section 5.1 | High |
| Failed sign-in threshold breach | 1 | Section 5.1 | High |
| Legacy authentication sign-in success | 1 | Section 5.1 | High |
| Bulk account operations | 1 | Section 5.1 | High |
| High-privilege application consent grant | 1 | Section 5.4 | High |
| ProtectionStatus non-reporting assets | 1 | Section 5.2 | High |
| Malware or threat detection (MDE integration) | 1 | Section 5.2 | Critical |
| Subscription Owner role assignment | 1 | Section 5.5 | Critical |
| Management port internet exposure | 1 | Section 5.3 | High |
| NSG rule modification in production | 1 | Section 5.3 | Medium |
| New service principal or application registration | 1 | Section 5.4 | Medium |
| Azure resource deletion in production | 1 | Section 5.5 | Medium |
| Defender for Cloud High or Critical alert | 1 | Section 5.2 | High |

### 6.2 Analytics Rule Standards

**Naming convention:** [Category] - [Specific Activity] - [Severity]. Example: Identity - Break-Glass Account Sign-In - Critical.

**Required rule configuration:** All rules must include: at least one MITRE ATT&CK tactic mapping, a description explaining what the rule detects and what the analyst should do when it fires, entity mapping for relevant entities (account, IP, host), and query lookback periods consistent with the detection objective.

**Incident creation:** All Critical and High severity rules must automatically create a Sentinel incident. Medium severity rules may be configured to alert only.

**Rule documentation:** A rule inventory document must be maintained listing all active rules, their detection objective, the date last reviewed, the last tuning action taken, and the false positive rate.

### 6.3 False Positive Management

Any rule generating more than 20 false positives per week must be tuned by the Detection Engineer within 10 business days. A rule that cannot be tuned to an acceptable false positive rate must be escalated to the Security Lead for a decision on suppression or retirement.

Suppression rules must be reviewed quarterly. Any suppression active for more than 90 days must be renewed explicitly or removed. Alert suppression may not be applied to Critical severity rules without Security Lead approval.

## 7. Alert Response Requirements

### 7.1 Alert Triage SLAs

**Driving Evidence (real finding):** At time of the March 2026 assessment, High and Critical alerts were present in the SecurityAlert table with no evidence of triage. The exact duration these alerts had gone unreviewed could not be determined from available telemetry. These SLAs exist to prevent that condition from recurring.

| Alert Severity | Triage SLA | Required Initial Action |
|---|---|---|
| Critical | 1 hour (any time of day) | Analyst reviews immediately; Security Lead notified; incident created; investigation begins |
| High | 4 hours (business hours); 8 hours (after hours) | Analyst reviews; incident created if not auto-created; investigation begins |
| Medium | 24 hours | Analyst reviews; incident created if warranted; documents disposition |
| Low | 72 hours | Analyst reviews; documents disposition |
| Informational | 5 business days (best effort) | Review when capacity allows; document disposition |

**After-hours Critical coverage:** A named on-call analyst must be designated for every week. The on-call analyst must respond to Critical alert notifications outside business hours within 1 hour.

### 7.2 Alert Severity Definitions

| Severity | Definition | Examples |
|---|---|---|
| Critical | Active compromise, imminent threat to a Tier 1 asset, or security control failure with immediate impact | Break-glass sign-in; active malware on Tier 1 asset; subscription Owner role assigned to unknown account; SFA success on Global Administrator |
| High | Likely threat, significant policy violation, or indicator of compromise requiring investigation | Failed sign-in threshold breached on privileged account; impossible travel; MFA disabled; new high-privilege consent grant |
| Medium | Potential policy violation or anomaly requiring review | New application registration; NSG rule modification; legacy authentication sign-in |
| Low | Noteworthy but unlikely to indicate active threat without additional context | Service principal sign-in from new location; resource tag modification |
| Informational | Logged for audit and compliance purposes with no implied threat | Successful PIM activation; password reset via SSPR; successful MFA registration |

### 7.3 Escalation Requirements

Any confirmed true positive Critical or High alert must be escalated to the Security Lead within 30 minutes of confirmation, regardless of time of day. Escalation must be via a documented real-time channel (phone, SMS, or designated secure messaging). Email alone is not sufficient for Critical or High escalations.

Any alert that cannot be triaged within its SLA window must be escalated to the Security Lead or on-call analyst immediately, not at the end of the SLA window.

### 7.4 Alert Closure Requirements

An alert may only be closed when one of the following conditions is documented:

| Closure Disposition | Required Documentation |
|---|---|
| True Positive -- Incident Created | Alert linked to the corresponding Sentinel incident |
| False Positive -- Rule Tuning Required | Reason documented; Detection Engineer notified for tuning |
| False Positive -- Expected Behavior | Specific authorized activity that triggered the alert documented |
| Benign True Positive | Activity confirmed authorized; documented rationale |
| Duplicate | Link to the original alert documented |

Closing an alert without documentation is a policy violation. Bulk-closing alerts without individual review is prohibited.

## 8. Incident Management Requirements

### 8.1 Incident Creation Policy

A Sentinel incident must be created for every confirmed or suspected true positive alert of Critical or High severity. Incidents created by analytics rules must not be deleted without review. If the triggering alert was a false positive, the incident must be closed with the appropriate false positive disposition and the tuning action documented.

**Driving Evidence (real finding):** At time of the March 2026 assessment, the SecurityIncident table contained unactioned incidents that had never been assigned an owner, investigated, or closed. This policy section establishes the mandatory incident management requirements to prevent that condition from recurring.

### 8.2 Incident Severity and Response SLAs

| Severity | Criteria | Initial Response SLA | Resolution Target |
|---|---|---|---|
| Severity 1 (Critical) | Confirmed or highly suspected active compromise of a Tier 1 asset or identity; active data exfiltration; Tier 1 unavailability due to security event | 30 minutes (any time of day) | 4 hours for initial containment |
| Severity 2 (High) | Confirmed compromise of a standard account or asset; active lateral movement; confirmed malware persistence | 1 hour (business hours); 2 hours (after hours) | 24 hours for initial containment |
| Severity 3 (Medium) | Suspected compromise under investigation; High alert confirmed true positive | 4 hours | 72 hours for initial containment |
| Severity 4 (Low) | Anomalous activity confirmed low-risk; policy violation with minimal security impact | 24 hours | 7 days |

### 8.3 Incident Escalation

All Severity 1 incidents must have an Incident Commander designated within 30 minutes of incident declaration. All Severity 1 and Severity 2 incidents must be communicated to senior leadership within 2 hours.

Any incident involving a Tier 0 asset (Global Administrator, break-glass account, or subscription Owner) is automatically Severity 1 regardless of initial assessment.

### 8.4 Incident Closure Requirements

An incident may only be closed when all of the following are met and documented:
1. Scope fully determined (all affected assets and accounts identified)
2. Root cause documented or documented as undetermined after investigation
3. Containment actions completed and verified
4. Evidence preserved and linked to the incident record
5. Post-incident summary written for all Severity 1 and Severity 2 incidents
6. Remediation actions documented in the incident record or POA&M
7. Incident reviewed by the Security Lead or their designee

## 9. Threat Intelligence Requirements

Microsoft Sentinel's built-in threat intelligence integration must be enabled. The Security Operations Analyst must review the Microsoft Security Intelligence Blog and CISA Current Activity feed at least weekly for threats relevant to Azure, Windows, Entra ID, and the operating sector.

Any vulnerability newly added to the CISA KEV Catalog that affects software present in the environment must trigger a review of existing Sentinel analytics rules. If no detection rule exists for a newly listed KEV vulnerability present in the environment, the Detection Engineer must create or enable a detection rule within 14 days of the KEV listing date.

## 10. Monitoring Effectiveness Requirements

### 10.1 Detection Coverage Assessment

The Detection Engineer must maintain a detection coverage matrix mapping active Sentinel analytics rules to MITRE ATT&CK tactics and techniques, reviewed and updated quarterly. High-risk coverage gaps must be addressed with a new detection rule within 60 days of identification.

### 10.2 Analytics Rule Effectiveness Review

All active analytics rules must be reviewed at least quarterly for effectiveness. A rule is considered ineffective if it has generated zero alerts in the past 90 days (potential logic error or data source issue), generated more than 20 false positives per week without tuning action, or has not been reviewed in more than 12 months. Ineffective rules must be tuned, replaced, or retired with documented rationale.

### 10.3 Monitoring Metrics and KPIs

| Metric | Target | Measurement |
|---|---|---|
| Mean time to detect (MTTD) | Less than 1 hour (Critical); less than 4 hours (High) | Time from event to alert generation in Sentinel |
| Mean time to triage (MTTT) | Within SLA per Section 7.1 | Time from alert creation to analyst first action |
| Mean time to resolve (MTTR) | Within SLA per Section 8.2 | Time from incident creation to closure |
| Analytics rule coverage | 100% of mandatory categories in Section 6.1 | Monthly rule inventory count |
| Alert false positive rate | Less than 30% of total alerts | (FP alerts / Total alerts) x 100 |
| Data connector availability | 99% uptime for Critical connectors | Connector health monitoring; heartbeat query |
| Unactioned alerts older than SLA | Zero | Count of alerts exceeding triage SLA with no analyst action |
| Unactioned incidents older than SLA | Zero | Count of incidents exceeding response SLA with no owner assigned |

**Driving Evidence (real finding):** At time of the March 2026 assessment, the last two metrics were non-zero. Unactioned alerts and unactioned incidents existed in the environment. These targets directly address that confirmed condition.

## 11. Security Monitoring Reporting

### 11.1 Weekly Security Monitoring Summary

Produced every Monday by 09:00. Required contents: total alerts by severity and source; alerts triaged within SLA versus outside SLA; true positive versus false positive count; incidents created, active, and closed; any Critical or Severity 1 events during the week; connector health status for all Critical connectors; any analytics rules added, modified, or retired.

### 11.2 Monthly Security Posture Report

Produced by the 10th business day of each month, submitted to the Security Lead and GRC Analyst. Required contents: monitoring metrics from Section 10.3 with trend line; summary of all Severity 1 and Severity 2 incidents; top 5 alert sources; top 5 false positive sources; data connector health summary; analytics rule inventory status; any monitoring coverage gaps; recommended actions for the following month.

### 11.3 Quarterly Detection Effectiveness Review

A quarterly review must cover: MITRE ATT&CK detection coverage matrix; analytics rule effectiveness review results; alert SLA compliance trend; incident response SLA compliance trend; false positive rate trend; data connector health; review of all active suppressions and exclusions; and improvement recommendations. The review must produce a written summary retained as audit evidence.

## 12. Privacy and Legal Requirements

All personnel with accounts in the environment must be informed that their activities are subject to security monitoring. This disclosure must be made in the acceptable use policy or user agreement signed at onboarding, in security awareness training, and in system banners on managed systems where technically feasible.

Security monitoring data in the Sentinel workspace must be treated as confidential. Access must be restricted to personnel with a documented operational need. Log data used as evidence must be preserved in its original form and must not be altered without Incident Commander approval.

When a legal hold is invoked, all security log data within scope must be preserved beyond normal retention schedules. Automated log deletion or archiving affecting held data must be suspended until the legal hold is lifted.

## 13. Exceptions and Waivers

Exception requests must include: the specific policy section that cannot be met, the technical or operational reason, the compensating control in place, the requested duration (maximum 90 days), and the approving authority.

All approved exceptions must be tracked in the POA&M and reviewed at each quarterly detection effectiveness review.

The following requirements may not be excepted under any circumstances:
- Section 4.1.3 (no alert or incident may be deleted without documented review)
- Section 7.1 (Critical alert triage within 1 hour)
- Section 8.3.1 (Incident Commander designation for Severity 1 within 30 minutes)
- Section 8.3.4 (Tier 0 asset incidents are automatically Severity 1)
